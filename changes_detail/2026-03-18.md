# Changes Detail - 2026-03-18

## Feature: Phase 1 Agent & Realtime Device Presence System

Implemented the foundational Device System (Phase 1) for a new event-driven, distributed download architecture. This system allows multiple remote devices to natively connect to HermesLink as worker nodes.

**Details:**
- **Agent Process Creation:** Built a standalone, lightweight daemon script (`backend/src/agent.py`) that acts as the core worker node. The agent automatically handles generating or loading persistent `device_id` identifiers upon startup so the device is uniquely tracked across reboots.
- **Firebase RTDB Integration:** Successfully integrated the agent with Firebase Realtime Database (RTDB) to manage live device presence without incurring heavy write costs (which Firestore would).
- **Heartbeat & Status Tracking:** Implemented a periodic 30-second heartbeat mechanism that actively pings the server with a `last_seen` timestamp. This bypasses the Python Admin SDK's lack of `on_disconnect()` support while effectively keeping the device's RTDB `status` locked as `"online"`.
- **Graceful Shutdown Hooks:** Handled manual/KeyboardInterrupt termination logic so the agent immediately sets its RTDB node state to `"offline"` upon closing.
- **Environment & Dependency Configuration:** Extended `backend/src/core/firebase_config.py` to seamlessly accept runtime `FIREBASE_DATABASE_URL` routing explicitly mapped from the local backend `.env`. Appended new requirements for `firebase-admin` and `psutil`.

## Planning: Event-Driven Architecture

- **Implementation Plans:** Created comprehensive implementation plans (`Planning/improved_plan.md` and `Planning/next_step.md`) outlining the transition to a new Event-Driven Architecture using Firebase RTDB and Firestore to support distributed remote download workers.
