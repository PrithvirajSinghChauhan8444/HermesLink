# Documentation of Changes – 2026-04-01

## 1. Custom Device Names via Config
* **Files Modified:** `backend/src/agent.py`, `backend/src/test_agent.py`
* **Details:** Replaced the simple `_get_or_create_device_id` method with a more comprehensive `_load_or_create_device_config` method. This allows the backend agents to read a `device_name` string from their respective JSON config files (`agent_config.json` and `test_agent_config.json`).
* **Fallback Behavior:** If `device_name` is absent from the configuration during boot, the agent defaults to the system hostname via `socket.gethostname()`. It then automatically populates and persists this default value back into the configuration file to allow for easy user modification in the future.

## 2. Optimistic UI Progress Animation
* **File Modified:** `frontend/HermesLink_frontend/src/components/sections/ActiveJobsSection.jsx`
* **Details:** Eliminated the rigid UI freezes during active downloads by introducing local progress extrapolation directly into the Active Jobs visual components.
* **Mechanism:** 
  - Added frontend parsing to automatically convert the raw `speed` string (e.g., `"1.5 MB/s"`) from Firebase back into raw bytes-per-second.
  - Deployed a frequent (150ms) local `setInterval` loop to optimistically advance the visual progress bar based on elapsed milliseconds multiplied by the current speed.
  - Implemented a strict visual growth cap of `+5%` above the authoritative Firebase state bounds to strictly prevent annoying regressive "backward snap" UI glitches when network payloads finally arrive.
  - Integrated an exponential speed decay curve triggering after 5.5 seconds of silence. This gracefully brakes the progress bar simulation instead of hitting a sudden wall if the backend sync stalls.

## 3. Increased Agent Polling Frequency
* **Files Modified:** `backend/src/test_agent.py`
* **Details:** Decreased the primary backend engine polling configuration (`MONITOR_INTERVAL_SECONDS`) from an excessive 30 seconds down to a much more responsive 5 seconds.
* **Outcome:** When combined with the new optimistic frontend rendering and the instant native Firebase RTDB hooks for state changes (Paused, Failed, Completed), the perceived system latency for end-user interaction sits near zero while maintaining network efficiency.
