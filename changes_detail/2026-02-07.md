## Architectural Research

### Key Question Explored
**Should HermesLink use a serverless or server-based architecture?**

### Research Findings

#### Server-Based Architecture (Recommended ✅)

After thorough analysis, a **server-based architecture with distributed local workers** was chosen:

| Component | Role |
|-----------|------|
| **Central Server** | Control plane — always running, manages coordination |
| **Local Devices** | Worker nodes — execute downloads locally |
| **Web Dashboard** | UI for job submission, monitoring, and device selection |

#### Why Serverless Was Rejected ❌

Serverless is **not suitable** for the core system because:
- Cannot maintain persistent connections
- Stateless by design
- Has execution time limits
- Cannot coordinate long-running or distributed jobs

> Serverless is only viable for **auxiliary tasks** like notifications, webhooks, or lightweight public APIs.

---

## Core Architecture Decisions

### 1. Central Control Server (Always-On)
Responsible for:
- Device registration and authentication
- Job and queue management
- Task assignment to specific devices
- Real-time status and progress tracking
- Handling device disconnects and reassignment

### 2. Local Devices (Worker Nodes)
Each device:
- Connects using a **persistent connection** (WebSocket/streaming)
- Waits for tasks to be assigned
- Executes downloads locally using native tools
- Reports progress, status, and results back to server

### 3. Communication Model
- Devices maintain **persistent connections**
- Server **pushes jobs instantly** (no polling required)
- Enables real-time updates similar to messaging platforms

---

## Device Job Retrieval Options

| Method | Pros | Cons |
|--------|------|------|
| **Polling** | Simple, NAT-safe, reliable | Slight latency |
| **Persistent Connection** | Real-time, better UX | More complex |

> Pure webhooks are **not reliable** for devices behind NAT.

---

## Recommended Build Order

1. Main server + DB
2. Job & device models
3. Polling-based agent
4. Web dashboard
5. Persistent connection upgrade (optional)
6. Notifications & polish

---

## Key Principles

- **Verify users, not devices**
- **Execution happens only where an agent runs**
- **UI never executes downloads**
- **Server is the source of truth**
- **Server never connects into devices — devices always connect out**

---

## Mental Model

> **Server assigns jobs → devices pull/receive → agents execute locally.**
