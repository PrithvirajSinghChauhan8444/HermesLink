# Changes Detail - 2026-03-02

## Fixed CORS Policy Error on `/api/queues`
- Frontend (running on `http://localhost:5173`) was being blocked by CORS when calling `http://localhost:8000/api/queues`.
- Added `CORSMiddleware` to the FastAPI backend (`api_server.py`) to include the correct `Access-Control-Allow-Origin` response header.
- `allow_origins`, `allow_methods`, and `allow_headers` were configured to permit requests from the frontend dev server.

## Investigated Active Jobs Data Flow
- Traced how active job data reaches the `ActiveJobsSection` component:
  - `useEffect` in `ActiveJobsSection.jsx` polls `endpoints.jobs.active()` every **1 second**.
  - `api.js` maps this to a `GET /api/jobs/active` axios call.
  - Backend filters jobs by `{PENDING, RUNNING, PAUSED}` states and returns them.

## Fixed: Failed/Stopped Jobs Persisting in Active Jobs List
- **Root Cause:** In `job_runner.py`, the `run_blocking()` wrapper for blocking engines (`media`, `direct`) only called `transition_job(..., FAILED)` on an exception. A clean engine finish (success) had no follow-up state transition, leaving the job permanently in `RUNNING` state and thus permanently visible in the Active Jobs list.
- **Fix applied to `job_runner.py`:**
  - On engine success → `transition_job(job_id, COMPLETED)` is now called.
  - On engine exception → `transition_job(job_id, FAILED, error)` is called inside its own try/catch to avoid swallowing the error.
  - Added a `finally` block to always clear `current_engine` and `current_job_id`, preventing the runner loop from holding stale references to a finished engine.

## Known Issue: Jobs Marked as RUNNING Without Engine Actually Starting (Active Jobs Misbehaving)
- Observed that some jobs transition to `RUNNING` state in the database but the underlying engine (download) never actually starts.
- This causes the Active Jobs section to display ghost "running" jobs — progress stays frozen, speed reads `0 B/s`, ETA shows `--:--:--`.
- The state is set by `process_queues()` → `transition_job(..., RUNNING)` before the `JobRunner` loop picks up and calls `_start_engine_for_job()`, so a timing gap or runner crash between those two moments can leave a job stranded in `RUNNING` with no associated engine thread.
- **Not yet resolved** — flagged for investigation into adding a engine-start confirmation step or a watchdog timeout that fails jobs stuck in `RUNNING` with no progress update.
